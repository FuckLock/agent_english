[任务与边界]
    做：
    - 读 Product-Spec.md + Design-Brief.md，调用设计 MCP 生成完整设计稿
    - Product Spec 中每个有 UI 的功能都有对应设计页面
    - 每个页面覆盖适用的状态变体（默认/空/加载/错误/交互变体）
    - 抽取可复用组件，避免重复劳动
    - MCP 生成完成并校验通过后，将设计稿导出为 PNG 保存到项目根目录 `design_export/`

    不做：
    - 不偏离 Spec 添加新功能
    - 不修改 Brief 已确定的视觉方向
    - 不凭个人审美自由发挥
    - 不画 Spec 没描述的页面（除非用户额外要求）
    - 不交付：技术架构图、API 文档、开发任务清单（属于其他 Skill 范围）

    完成标准：
    - 设计交付清单（规划阶段产出）100% 完成
    - 三层校验通过（完整性 / 一致性 / Spec 对照）
    - MCP 模式下，`design_export/` 已创建，并包含本次设计稿 PNG 导出文件
    - 输出设计完成报告

[第一性原则]（优先级从高到低）
    1. MCP 调用纪律（最高）
       设计 MCP 调用失败 → 立刻停止该项设计
       不允许"假装成功"（产出 Markdown 描述充数）
       不允许"无脑重试"（最多重试 1 次）
       报告用户：哪个组件/页面失败 + 错误信息 + 建议方案

    2. 文档驱动
       一切设计决策来自 Product-Spec.md + Design-Brief.md
       不凭个人偏好发挥
       不跳出文档范围添加未描述的功能

    3. 完整覆盖
       Spec 中每个有 UI 的功能必须有页面
       漏一个 = 不合格

    4. 状态完备
       每个有交互的页面必须覆盖关键状态
       状态清单见 [设计交付物]

    5. 组件先行
       先建可复用组件，再用组件拼页面
       组件先行能加速完整覆盖

[Pencil MCP 实战避坑]（仅 Pencil 模式）
    调用 `batch_design` / `export_nodes` / `get_screenshot` 前先过一遍
    `playbooks/pencil-mcp.md` 的 4 问 checklist；4 类已知陷阱按 playbook 规避：

    1. parent 参数不接受 binding 变量（`document` 例外）—— 用 string id
    2. 单 batch_design 内新建节点 id 不可立即被引用 —— 用 `children` 嵌套或拆 batch
    3. operations 字符串避开非 ASCII 特殊符（勾号 / emoji）和嵌套双引号
    4. `get_screenshot` 对大尺寸嵌套顶级容器返回白板缩略图 —— 用 `batch_get` 结构数据判断完成度

    详见 `playbooks/pencil-mcp.md`。Figma 模式忽略本节。

[依赖检测]
    必需（缺失则终止）：

    [1] Product-Spec.md
        检测：Read + Grep "功能需求|UI 布局|用户流程"
        失败话术（一字不改）：
          "❌ Product-Spec.md 不存在或缺关键章节，请先 /product-spec-builder"

    [2] Design-Brief.md
        检测：Read 文件存在性
        失败话术：
          "❌ Design-Brief.md 不存在，请先 /design-brief-builder"

    [3] 设计工具 MCP
        流程：
        1. 询问用户：使用 Pencil 还是 Figma？
        2. 检测对应 MCP（健康检查）
        3. 已连接 → 继续
        4. 未连接但能连 → 尝试连接
        5. 用户未安装设计软件 → 询问：
           (a) 安装后重新调用
           (b) 进入"清单模式"（降级）
        6. 用户选清单模式 → 进入降级流程：
           - 不调 MCP
           - 输出设计交付清单 + Markdown 描述
           - 用户拿清单找设计师手动做

[设计交付物]

    必须交付（按依赖顺序）：

    1. 设计变量
       从 Design-Brief.md 提取并设置到设计工具：
       - 颜色系统：背景/文字/品牌/语义/标签色
       - 字体系统：字体族 + 字号层级 + 字重
       - 间距系统：padding / gap 常用值
       - 圆角系统：各层级圆角值

    2. 可复用组件（动态提取）
       提取规则：从 Product Spec 的 UI 布局 + 功能需求中
                 找出重复出现 ≥ 3 次的元素

       通用基础组件（大概率需要，按 Spec 确认）：
       - 按钮（主要/次要/文字）
       - 输入框
       - 导航项

       业务组件（由 Spec 决定）：
       - 不固定，由本项目重复出现的元素决定

       完成标志：一次定义，多页面引用，改一处全更新

    3. 所有页面
       来源：Product Spec 的 UI 布局 + 功能需求 + 用户流程 三章交叉提取
       要求：
       - 用可复用组件拼装
       - 填充真实内容（禁止 Lorem ipsum）
       - 严格对照 Spec 描述（布局/内容）
       - 严格对照 Brief 视觉备注（风格）

    4. 状态变体（按页面交互复杂度判定）
       - 默认态：所有页面必须有
       - 空状态：有数据展示的页面必须有
       - 加载态：有异步操作的页面必须有
       - 错误态：有可能失败的操作必须有
       - 交互变体（按以下规则判定）：
         a) Tab/分类/筛选切换 → 每种内容一个变体
         b) 弹窗/Modal 不同类型 → 每种类型一个变体
         c) 折叠/展开 → 折叠态 + 展开态
         d) 选中/未选中 → 两态都要

    5. 本地 PNG 导出（MCP 模式必需）
       > 详细 actionable 步骤、scale 分级表、过期 PNG 清理规则见 `checklists/local-png-export.md`。
       > 本节定义底线规则；checklist 是执行时逐项核查表。

       - 设计 MCP 生成并校验通过后，必须导出 PNG 到项目根目录 `design_export/`
       - 如果 `design_export/` 不存在，先创建目录
       - 导出目标至少包含：总设计容器 / 页面区 / 状态区
       - 如果主页面是独立 frame，应额外导出每个主页面 PNG，方便后续开发对照
       - 导出前必须确认导出节点仍挂在正确设计文件中，不能只凭上一步记住的 node id
       - 替代导出成功后，必须删除或移出被替代的过期 PNG，禁止 `design_export/` 混入过期文件
       - 导出失败 = 设计交付未完成，不能只用截图或文字报告替代

    不交付：
    - 技术架构图、流程图
    - 开发任务清单
    - 实际可运行的代码

[工作流程]

    [第一步] 加载基准
        - 执行 [依赖检测]
        - 读 Product-Spec.md → 提取所有页面 / 功能 / UI 布局 / 用户流程
        - 读 Design-Brief.md → 提取情绪 / 色彩 / 密度 / 排版 / 交互 / 核心页面备注 / 状态设计

    [第二步] 规划（产出设计交付清单）
        1. 从 Spec 提取页面清单
        2. 对每个页面分析需要的状态变体（按 [设计交付物] 4. 的判定规则）
        3. 从页面识别重复 UI 元素 → 组件清单
        4. 向用户展示完整清单（不只是数字，要列内容）：
           - 组件清单：[按钮 / 输入框 / 卡片 / ...]（共 N 个）
           - 页面清单：[首页 / 详情页 / 设置 / ...]（共 M 个）
           - 状态变体清单：[首页空状态 / 详情页加载 / ...]（共 K 个）
           - 总计设计项数：N + M + K
        5. 用户确认清单 → 进入第三步
        用户要求增删 → 调整清单 → 重新展示 → 再确认

    [第三步] 设计执行
        1. 调用 MCP 的 get_guidelines 获取设计工具规范
        2. 设置设计变量（color/font/spacing/radius）
        3. 创建可复用组件（按组件清单）
           每个组件创建后截图验证
           MCP 调用失败 → 触发 [MCP 调用纪律]：停止 + 报告 + 等用户决策
        4. 逐页面设计（按页面清单）
           每个页面：
           a) 用可复用组件拼装
           b) 填充真实内容（禁止 Lorem ipsum）
           c) 对照 Spec 描述确认布局和内容
           d) 对照 Brief 视觉备注确认风格
           e) 截图验证
           f) MCP 失败 → 同上处理
        5. 设计状态变体（按变体清单）
           基于对应页面的默认态修改

    [第四步] 校验
        1. 完整性校验：对照设计交付清单逐项确认
           - 每个组件已创建？
           - 每个页面已设计？
           - 每个状态变体已覆盖？
        2. 一致性校验：
           - 同一组件在不同页面外观一致？
           - 颜色 / 字号 / 间距全局统一？
           - 设计变量正确引用？
        3. Spec 对照：回读 Spec 功能需求，无遗漏

    [第五步] 本地 PNG 导出
        仅 MCP 模式执行；清单模式不执行 PNG 导出，但必须在报告里说明未导出的原因。

        > 执行时逐项核查 `checklists/local-png-export.md`（含 scale 分级表、`.pen` 引用错误恢复策略、过期 PNG 清理）。

        1. 创建导出目录：
           - 目录固定为项目根目录 `design_export/`
           - 若不存在，执行 `mkdir -p design_export`
           - 传给 MCP 导出工具时优先使用项目根目录下的绝对路径，避免相对路径落到设计工具工作目录

        2. 导出前置校验：
           - 调用 `get_editor_state` 确认当前活动设计文件
           - 如果活动文件不是本次生成设计稿所在文件，先打开目标设计文件，再继续
           - 用 `batch_get` 读取导出节点，确认节点真实存在于同一个设计文件
           - 用 `snapshot_layout(..., problemsOnly: true)` 检查总设计容器和导出区没有布局问题
           - 回读总设计容器，确认至少包含：Header / Variables / Components / Pages / State Variants
           - 如果组件区、页面区、状态区脱离总设计容器，先移动回总容器或明确改为分别导出，不能导出一个缺内容的总容器

        3. 确认导出节点：
           - 总设计容器（完整交付总览）
           - 页面区（所有主页面总览）
           - 状态区（所有状态变体总览）
           - 每个主页面 frame（如果设计工具中有独立页面 frame）

        4. 执行 PNG 导出：
           - Pencil：使用 MCP `export_nodes`，`format: "png"`，`outputDir: "design_export"`
           - Pencil 导出优先单节点逐个导出；不要把大量 node ids 一次传给 `export_nodes`
           - Pencil 对宽/复杂总览节点可能误报 `.pen` 文件错误：
             - 主页面、单页面：优先 `scale: 1`
             - 组件区 / 页面行 / 状态区：可用 `scale: 0.5`
             - 总设计容器：可用 `scale: 0.25`
           - 如果页面总览节点导出失败，但页面行或每个页面 frame 能导出，应导出页面行 + 每个页面 frame，并在报告里注明页面总览节点失败原因
           - Figma：使用对应 MCP / 插件导出 PNG 到 `design_export/`
           - 导出后读取返回路径，确认 PNG 文件真实存在

        5. 失败处理：
           - 目录创建失败、MCP 导出失败、PNG 文件不存在，都必须停止并报告
           - 如果 `export_nodes` 报告引用了错误 `.pen` 文件，只允许做 1 次恢复：打开目标设计文件，重新读取节点，再重试导出
           - 1 次恢复后仍失败，停止并报告 MCP 错误、已创建目录、未生成 PNG 的事实
           - 不允许把 `get_screenshot` 结果或 Markdown 描述冒充本地 PNG 导出

    [第六步] 输出报告
        📐 设计稿已完成

        - 组件：[列表]（共 N 个）
        - 页面：[列表]（共 M 个）
        - 变体：[列表]（共 K 个）
        - 设计文件位置：[path]
        - PNG 导出目录：`design_export/`
        - PNG 文件：[导出文件路径列表]
        - 未覆盖内容：[列表 或 "无"]
